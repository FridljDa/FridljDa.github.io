## Lora
- A fine-tuned model's weight matrix W' ≈ W₀ + ΔW, where ΔW has low intrinsic rank (r).
- The modified forward pass is h = W₀x + BAx, where B ∈ ℝ^(d×r), A ∈ ℝ^(r×k), and rank r ≪ min(d,k). A is initialized with random Gaussian, B is initialized to zero — so ΔW = BA is zero at the start of training.
- In practice, we scale:

$$h = W_0x + \frac{\alpha}{r} \cdot BAx$$

The **α/r scaling** is applied to the adapter's contribution before it's added to the frozen weights' output. +
By dividing by r, we normalize the adapter's output based on its rank, ensuring that the magnitude of the updates remains consistent regardless of the chosen rank.

Think of it this way. ΔW = BA is a sum of r rank-1 terms:

$$\Delta W = \sum_{i=1}^{r} b_i a_i^T$$

## The core idea: trading range vs. precision

A floating point number has three parts:

```
[sign | exponent | mantissa]
```

- **Exponent** → how big/small the number can be (range)
- **Mantissa** → how precisely you can represent it (precision)

You have a fixed bit budget. Every format makes a different tradeoff.

## Why the adapters A and B must stay in bf16

During backprop, gradients of A and B can be very small (especially early in training) or vary widely in magnitude across layers. You need:
- Enough **range** to not overflow (→ need 8 exponent bits, rules out fp16)
- Enough **precision** to accumulate small gradient updates meaningfully (→ rules out 4-bit)

bf16 is the minimum viable format for this.

---

## Why W₀ can drop to 4-bit

W₀ is frozen — it never receives gradient updates. It just needs to produce a numerically reasonable matrix multiplication output. Pretrained weights are well-behaved (approximately Gaussian, no extreme values), so you can quantize aggressively.
The forward pass accumulation still happens in bf16 — you dequantize block-by-block on the fly during the forward pass, multiply, then discard. You never store a bf16 copy of the full W₀.

---

## The full QLoRA forward pass, concretely

```
1. Load W₀ block in NF4 (4-bit, on GPU)
2. Dequantize that block → bf16 (temporary, small)
3. Compute W₀x in bf16
4. Discard the bf16 block
5. Add (α/r) · BAx  [A, B already in bf16]
6. Output h in bf16
```

The memory saving comes from step 1: you never have a full bf16 W₀ in VRAM simultaneously.

---

## One-line intuitions to remember

- **fp32**: the "safe default," expensive, used for optimizer states in sensitive cases
- **bf16**: fp32's range, half the memory — the standard training format on modern GPUs
- **fp16**: smaller range than bf16, mostly legacy, avoid for training
- **NF4**: not a general float format — a lookup table optimized for Gaussian weight distributions, inference/frozen weights only

The quantization/dequantization process
python# Quantization (once, before training)
index = argmin(|w - codebook|)  # find nearest of 16 levels
store(index)                     # 4 bits

# Dequantization (every forward pass, block by block)
w_approx = codebook[index]      # lookup, gives bf16 value

One more detail: block-wise scaling
The codebook values are in [-1, 1]. Real weights aren't. So before quantizing, each block of 64 weights gets divided by its own absmax scalar to normalize into [-1, 1]. That scalar is stored separately (in fp32, then itself double-quantized).
So the full stored representation per weight is:

4-bit index (the weight itself)
A shared absmax scalar per 64-weight block

This is why it's not just a lookup table — it's a lookup table plus a per-block scale factor.

# speculative decoding

Great topic — it's one of the most practically important inference-time optimizations and comes up a lot in production LLM work.

---

## The core problem it solves

Autoregressive generation is **sequential by construction** — to generate token t+1 you need token t. This means you can't parallelize across the output sequence. Each forward pass through a 7B+ model costs real time, even on good hardware.

Speculative decoding breaks this bottleneck.

---

## The idea

You use **two models**:

- **Draft model**: small, cheap (e.g. 1B)
- **Target model**: large, expensive (e.g. 70B) — the one you actually care about

The algorithm:

```
1. Draft model generates k tokens quickly (e.g. k=5)
2. Target model evaluates all k tokens in ONE forward pass
3. Accept or reject each draft token based on probability ratio
4. If rejected at position i, resample from target at that position
5. Repeat
```

Step 2 is the key insight — the target model can score k tokens **in parallel** in a single forward pass because at verification time you already have the full sequence. You're exploiting the fact that the expensive model's forward pass over a known sequence is much cheaper than generating autoregressively.

---

## The math (important for interviews)

Let the draft model propose token x with probability q(x), and the target model assign probability p(x).

Accept x with probability:

$$\min\left(1, \frac{p(x)}{q(x)}\right)$$

If p(x) ≥ q(x) — target agrees or is even more confident → **always accept**

If p(x) < q(x) — draft was overconfident → **accept with probability p/q, otherwise resample**

This is rejection sampling. The crucial guarantee: **the output distribution is exactly the target model's distribution**. Speculative decoding is lossless — not an approximation.

---

## Why it works in practice

Most tokens in fluent text are predictable. "The cat sat on the ___" — even a small model will draft "mat" or "floor" correctly. The target model spends its budget verifying cheap tokens in bulk and only does real work when the draft goes wrong.

Typical speedup: **2–3× wall-clock time** for the target model, with zero quality degradation.

---

## The failure mode

If the draft model is too different from the target — wrong domain, wrong task — rejection rate is high and you get little benefit. The draft model needs to be a **good approximation** of the target, just cheaper. Common choices:

- A smaller model from the same family (e.g. DeepSeek-1.3B drafting for DeepSeek-67B)
- An earlier layer checkpoint of the same model
- The target model itself with early exit (self-speculative decoding)

---

## Connection to your work

For your cinema SaaS customer support system, if you fine-tune DeepSeek with QLoRA, speculative decoding is a natural complement: fine-tune both the large target **and** a small draft model on your domain data, then use speculative decoding at inference. The draft model will have high acceptance rate because it's in-domain. This is one of the cleaner ways to get near-full-model quality at significantly reduced latency cost.

---

## Interview one-liner

> "Speculative decoding uses a cheap draft model to propose k tokens, then verifies all k in one parallel forward pass of the expensive target model using rejection sampling with acceptance probability min(1, p/q). Output distribution is provably identical to the target model alone — it's a pure latency optimization, not an approximation."
